1. TL;DR — 60-second read

The build: a daily GitHub Actions cron (08:00 UTC) that runs the existing v5 pipeline end-to-end + three new pieces — GSC discovery, Rank Math 90+ scorer, and a publish-decision gate that auto-publishes when score≥90 / drafts when below.

Why GH Actions and not Claude routines: all 6 existing crons already run there, Anthropic SDK + Python is reliable for scheduled jobs, and MCPs are flaky in cloud-routine environments. Local MCP path is kept for human-triggered overrides.

Why "extend v5" not "rebuild": `tools/content/` already has discover, cannibalization, draft_blog, rank_math, push_wordpress, inventory. We add GSC source + scoring loop + daily orchestrator. ~3–4 days of new work, not 2–3 weeks.

What gets removed: image pipeline (`image_picker.py` skipped in this lane). Nikki gate becomes optional toggle, defaults OFF for daily auto-publish, ON for weekly Josh-review batch.

What I need from you before I build: 6 decisions in §11 — biggest are (a) auto-publish vs draft-only, (b) GH Actions vs Claude routine, (c) Sunday/Wednesday-style topical clustering vs flat daily. Everything else has a sensible default.

4. Gaps to close (what's missing in v5 for "daily auto-publish")

Gap 1 · Google Search Console MCP not installed

GSC is the highest-leverage signal (striking-distance keywords pos 11–30 = pickable wins). Today's pipeline can read GSC indirectly via Searchable, but a native MCP is faster + cheaper + more granular.

Fix: install @modelcontextprotocol/server-google-search-console or build a thin Python wrapper around the GSC API + service-account auth. Use the existing GA4 service account (project n8n2o-455819) — GSC is part of the same Google org.

Gap 2 · No daily orchestrator

v5's batch_generate.py orchestrates Sunday (discover) and Wednesday (publish) as separate triggers. Daily mode collapses both into one orchestrator that runs end-to-end every day.

Fix: new tools/content/daily_orchestrator.py that wraps the existing 7 stages into a single idempotent run. New cron .github/workflows/blog-daily.yml at 08:00 UTC.

Gap 3 · No Rank Math 90+ scorer (pre-publish gate)

v5 writes Rank Math meta fields but doesn't score the draft against Rank Math's 100-point checklist before publish. Daily mode needs this gate because there's no Nikki/Josh review.

Fix: new tools/content/rank_math_scorer.py that replicates the public Rank Math algorithm (focus-keyword density, title format, slug, meta description, content length, headings, links, images N/A, etc.) and returns a 0–100 score. Threshold for auto-publish: ≥90. Below 90 → save as draft + slack alert.

Gap 4 · No publish-decision gate

Today every blog passes through Nikki → Josh. Daily mode replaces the human gate with an algorithmic gate (score-based) plus a weekly summary digest so Josh still has oversight without being a bottleneck.

Fix: new tools/content/publish_gate.py with three policies: auto_publish (score≥90, no flags), draft_queue (anything below threshold or flagged), force_review (regulated terms, competitor mentions, >$25K-claim assertions).

Gap 5 · No ClickUp integration in v5

Title-approval uses ClickUp checklists today but only at the title-slate stage. Daily mode needs a per-blog ClickUp task that tracks its lifecycle (discovered → drafted → published → tracked).

Fix: new tools/content/clickup_sync.py — single function `create_or_update_blog_task(slug, status, links)`. Lightweight; not a hard dependency (if ClickUp is down, blog still ships, task is reconciled on next run).

Gap 6 · Inventory snapshot is stale (10d)

The cannibalization layer fails-closed when inventory is >7d old. content-inventory-refresh.yml exists but hasn't been wired to a daily cron.

Fix: change the cron expression to daily 05:00 UTC (3 hours before the blog cron). One-line YAML edit. Already on the BLOCKERS list.

  • FAQPage + BlogPosting JSON-LD injection (in draft_blog.py)

Scheduling layer — three options, with recommendation

Recommended

A. GitHub Actions daily cron

How: .github/workflows/blog-daily.yml @ 08:00 UTC. Python orchestrator calls REST APIs directly (GSC, GA4, Searchable, WP) + Anthropic SDK for writes.

Pros: proven (6 existing crons), zero new infra, easy debugging via Actions UI, no cost.

Cons: no native MCP — must wrap each integration in Python. Already done for most.

Fallback if down: manual local run via python3 -m tools.content.daily_orchestrator in any Claude Code session.

B. Claude routine (cloud schedule)

How: /schedule skill creates a daily routine that invokes a Claude Code session with MCP tool access.

Pros: uses real MCPs (no REST wrapping). Better failure messages because Claude reasons over them.

Cons: cloud-routine MCP support is still maturing; auth refresh (GA4 ADC) won't work in cloud — needs service account migration. Per-run cost.

Use case: a secondary monitoring routine that reads yesterday's published blog and writes a quality summary. Not the primary writer.

C. n8n daily workflow

How: daily-cron node in n8n triggers HTTP requests to GSC + GA4 + Anthropic + WordPress.

Pros: visual editor — easy to inspect mid-flow. Built-in Google nodes.

Cons: CLAUDE.md rule #22 says Inngest is the planned runtime; n8n stays as business-logic only. Adding a content workflow here muddies the boundary. Also elestio-n8n MCP validation must run before deploy.

Use case: not recommended for primary; could host the publish-success Slack notification.

Hybrid recommendation: primary writer = GitHub Actions (A). Monitoring/digest = optional Claude routine (B). Slack alerts = n8n (C) if you want them. This matches how the rest of your stack already works.

All times UTC. Total wall clock ~6–10 minutes. Each stage is idempotent — re-running mid-failure resumes from last completed stage via the queue file.

REFRESH
05:00 — inventory snapshot pulls WP posts. Fail-closed if stale.
2
DISCOVER
08:00 — GSC striking-distance + GA4 underperformers + Searchable 0%-prompts + Fathom objections. ~30 candidate topics.
3
CANNIBALIZE
Inventory check — drop any candidate that overlaps an existing slug/focus-keyword/canonical.
4
SLATE + PICK
Generate 12 titles, score, pick top 1. Single topic for today.
5
BRIEF
Outline + LSI keywords + FAQ-Q list + internal-link plan + external-authority plan + Rank Math target.
6
DRAFT
Claude writes via brand-voice skill. Inject JSON-LD. Insert links. ~1,500–2,200 words.
7
SCORE GATE

8
PUBLISH + TRACK
WP publish via JSON-RPC. NitroPack purge. ClickUp task closed. T+7/14/30 tracker scheduled.

Per-stage detail

Stage 1 · REFRESH (idempotent · 60s)

Runs at 05:00 UTC via content-inventory-refresh.yml (existing). Pulls all published posts in categories 27/28/29/30/13/14/10. Writes data/content/inventory/firstmovers-ai.json. Auto-commits with chore(content-inventory): refresh [skip ci].

Stage 2 · DISCOVER (parallel · 90–180s)

4 parallel sources, each returning ~10 candidates with priority score:

  • GSC striking-distance — keywords at pos 11–30 with >100 impressions/30d. Each → an "improve existing OR write supporting post" decision.
  • GA4 underperformers — pages with declining sessions or rising bounce. Each → a "refresh" candidate.
  • Searchable 0% prompts — the 11 non-branded prompts at 0% visibility. Each → an "AEO-targeted" net-new post.
  • Fathom objections — recurring prospect questions from sales-vault extractions. Each → a "FAQ-style" post that doubles as sales enablement.

Output: data/content/queue/YYYY-MM-DD-candidates.json with ~30 ranked candidates.

Stage 3 · CANNIBALIZE (10s)

For each candidate, check: slug overlap, focus keyword exact match, canonical conflict, title 70% similarity. Existing logic in cannibalization.py. Drop any candidate that fails. Typical drop rate: ~20–30%.

Stage 4 · SLATE + PICK (60s + Claude write)

title_slate.py generates 12 titles per surviving candidate. Score by CTR signals (number, year, "how to", "vs", question form), Rank Math title format, length 50–60 chars. Pick top topic + top title. One topic per day — no double-publish.

Stage 5 · BRIEF (60s)

Build BlogBrief:

{
  "title": "...",
  "focus_keyword": "...",
  "secondary_keywords": [10x LSI from GSC related queries],
  "outline": [H2/H3 hierarchy, 8–12 sections],
  "faq_questions": [5–8, phrased to match Searchable prompts],
  "internal_links": [{"to": "/consulting/", "anchor": "..."}, ...],
  "external_links": [3–5 authority sources],
  "cta_block": "DFY→/consulting/" | "DIY→/labs/",
  "category": 27|28|29|30,
  "rank_math_target": 90
}

Stage 6 · DRAFT (Claude write · ~2–3 min)

Anthropic SDK call (Claude Sonnet 4.6 or Opus 4.7 — see §11.4) with the brief + brand-voice skill loaded. Writes 1,500–2,200 words. Returns HTML body. Post-processing: inject JSON-LD blocks, inject internal/external links via the existing injectors, inject FAQ accordion if Elementor-shaped or plain <details> if classic editor.

Stage 7 · SCORE GATE (10s · the load-bearing decision)

rank_math_scorer.py computes the 0–100 score against Rank Math's public criteria. See §8 for the checklist. Three branches:

  • Score ≥90, no flags → status=`publish`
  • Score 80–89 OR minor flag → status=`draft`, append to data/content/queue/needs-review.json, post to ClickUp + Slack
  • Score <80 OR critical flag (regulated term, "free audit" slip, >$ claim) → status=`block`, abort publish, alert

Stage 8 · PUBLISH + TRACK (30s)

If `publish`: JSON-RPC to wpmcp/streamable, status=`publish`, Author=Josh McCoy. NitroPack purge via WP REST. ClickUp task → status `Published`. Schedule T+7/14/30 tracker entry in data/content/track/queue.json (the existing refresh.py picks these up).

If `draft`: same JSON-RPC but status=`draft`. ClickUp task → status `Needs Review`. Slack message to Josh's review channel.

8. Rank Math 90+ checklist (the scoring gate)

Rank Math's public scoring algorithm. Each item is checked by rank_math_scorer.py before publish. Score breakdown matches Rank Math's "Content AI"-disabled scoring (which we use).

Basic SEO (40 pts max)

  • Focus keyword in title (5)
  • Focus keyword in meta description (4)
  • Focus keyword in URL slug (5)
  • Focus keyword in first 10% of content (5)
  • Focus keyword in content (≥3 times, density 1–2.5%) (3)
  • Title under 60 chars (3)
  • Meta description 120–160 chars (3)
  • URL contains hyphens, no stop words, <75 chars (2)
  • Image alt text — SKIPPED (no images)
  • External authority link present (5)
  • Internal links (≥2) present (5)

Additional (35 pts max)

  • Focus keyword in H2 or H3 (3)
  • Focus keyword density 1–2.5% (3)
  • Content length ≥1500 words (5)
  • Multiple H2 / H3 headings (3)
  • Short paragraphs (max 120 words each) (3)
  • Use of <table>, <ul>, <ol> (5)
  • Power words in title (e.g., "Ultimate", "Proven", "How") (3)
  • Numbers in title (3)
  • Sentiment in title (positive/negative) (3)
  • Linking to high-DA external (3)
  • FAQPage JSON-LD present (1)

Title readability (25 pts max)

  • Title contains focus keyword (5)
  • Focus keyword near start (3)
  • Title has power word (3)
  • Title has number (3)
  • Title length 50–60 (3)
  • Flesch reading ease ≥60 (5)
  • No "free audit" / banned phrase (3, hard-fail if present)

Expected score for our drafts

If every existing module behaves: 87–94 per blog. Misses 1–2 items per blog typically (image alt is auto-zero; sometimes density 0.9% instead of 1%). Threshold for auto-publish: ≥90. 80–89 → draft. <80 → block.

Note: the Rank Math scorer is an approximation of Rank Math's plugin score. Some metrics (Flesch reading ease) require shipping a Python implementation; others (linking to external) are trivial. The implementation matches the plugin's open-source scoring code to within ±2 points in v5's regression tests.

9. Anti-cannibalization (multi-layer defense)

Cannibalization is your hard-stated #1 concern. Four layers, each independently blocks a topic if it overlaps published content.

  1. Layer 1 · Slug exact match. If the proposed slug already exists in inventory.json, drop instantly. Cheapest check.
  2. Layer 2 · Focus keyword exact match. Compare against every published post's rank_math_focus_keyword. If matched, drop.
  3. Layer 3 · Title similarity ≥70%. Use difflib SequenceMatcher. Two posts with "AI workflow automation for small business" and "AI workflow automation: small business guide" would match.
  4. Layer 4 · Canonical conflict. If the candidate's canonical URL would equal an existing canonical, drop. Catches edge cases where slug differs but URL structure converges.

What "drop" means: the candidate is logged in data/content/queue/dropped.json with reason. Next-best candidate from the day's slate is promoted. If <5 surviving candidates after dedup, the orchestrator emits a Slack alert and skips publishing that day (better than publishing a thin/duplicate post).

Edge case: "refresh" vs "new"

If GSC says we're at position 14 for "AI consulting" — there's already a /consulting/ landing page. The right answer isn't a duplicate blog; it's a refresh of the existing page. The orchestrator classifies these as refresh_candidates and writes them to a separate queue that refresh.py consumes weekly. Daily mode publishes new posts only — refreshes go through Josh's weekly review.

10. Brand voice rubric (already shipped, codified here)

Loaded at write-time via the firstmovers-blog-rubric Skill. Reproduced here for the plan so you can audit.

Tone

  • Operator-to-operator. Not vendor-to-prospect.
  • Specific over generic. Numbers, names, dates.
  • Show, don't sell. Examples > claims.
  • No corporate copy ("we're passionate about…").
  • No exclamation points. Period.

CTA framework (CLAUDE.md)

  • Never "free audit" — consulting is $25K–$60K
  • DFY intent → /consulting/ "Schedule a Free Strategy Call"
  • DIY intent → /labs/ "Explore AI Labs"

Structure

  • 1 H1, no period at end
  • 4–8 H2 sections, 0–3 H3 per H2
  • Short paragraphs (max 3–4 sentences)
  • One table, one list per ~600 words (Rank Math signal)
  • 5–8 FAQ at bottom, JSON-LD'd
  • Author = Josh McCoy
  • Categories 27/28/29/30 (only) for daily-mode posts

Forbidden phrases

"free audit", "skyrocket", "game-changer", "in today's fast-paced world", "leverage" (as a verb in marketing context), "synergy", emoji

11. Decisions you owe before I write code

Each below has a default I'd ship if you say nothing. Mark in your reply only the ones you want different.

Decision 11.1

Auto-publish or draft-only?

Default: auto-publish when Rank Math ≥90, no flags, no cannibalization. Below 90 → draft queue + Slack alert. Critical flags (regulated term, banned phrase, >$ claim) → block + alert.

Alternatives: (a) always draft, you publish manually each morning. (b) auto-publish only Tier-2 categories (Tools / AGI / In Business), keep Tier-1 (Consulting / Automation / Sales / Marketing) in draft.

Decision 11.2

Scheduling layer — GH Actions vs Claude routine?

Default: GH Actions cron at 08:00 UTC (= 04:00 ET = 12:00 GST). Same pattern as the 6 existing crons. Optional Claude routine for monitoring/digest only.

Alternative: Claude routine as primary, GH Actions as cold backup. Requires service-account migration for GA4 (ADC won't work in cloud routine).

Decision 11.3

Topical clustering — flat daily vs themed days?

Default: flat daily — pick the highest-priority candidate each day regardless of category.

Alternative: themed week (Mon = Consulting, Tue = Automation, Wed = Sales, Thu = Marketing, Fri = AI Tools, Sat = AGI, Sun = In Business). More predictable for SEO clustering but locks the orchestrator into picking from a smaller pool per day.

Decision 11.4

Claude model — Sonnet 4.6 or Opus 4.7?

Default: Sonnet 4.6 for the daily write (~$0.20–0.40/blog, 1500–2200 words). Opus 4.7 only for Tier-1 categories or when Rank Math first-pass scores <85 and we want a rewrite.

Alternative: Opus 4.7 always (~$1.20–2.00/blog) — depth and brand voice both improve. Worth it if you want 0% bounce on Josh-quality threshold.

Decision 11.5

Inventory refresh — daily cron vs current weekly

Default: change content-inventory-refresh.yml cron to daily 05:00 UTC. Removes the staleness blocker permanently. One-line YAML edit.

Alternative: keep weekly but force-refresh inside the daily orchestrator (Stage 1). Slower per-run but no separate cron to maintain.

Decision 11.6

GSC MCP install vs Python adapter only?

Default: Python adapter only (in GH Actions). Same code can be wrapped as an MCP later if you want it for local sessions. ~80 LOC, half a day's work.

Alternative: install official GSC MCP server too. Doubles maintenance surface but unblocks local Claude Code sessions to query GSC directly.

Decision 11.7

ClickUp lifecycle tracking — on or off?

Default: ON — one task per blog with status transitions. Soft dependency, won't block publish if ClickUp is down.

Alternative: OFF — just use the local review_queue.py file. Simpler but less visible to non-engineers.

13. Scenario walkthroughs

Scenario A · Happy path (the common case, ~80% of days)

  1. 05:00 UTC: inventory cron runs, refreshes 13 categories. ✓
  2. 08:00 UTC: daily-blog cron starts.
  3. Discover pulls 32 candidates across 4 sources.
  4. Cannibalization drops 9 (overlap with existing slugs). 23 survive.
  5. Slate picks "How AI workflow automation cuts back-office hours by 60% (2026 guide)" from the GSC striking-distance pool.
  6. Brief: focus_keyword=`ai workflow automation`, outline 9 sections, FAQ 6 Qs.
  7. Sonnet 4.6 writes 1,850 words in 90s.
  8. Rank Math scorer returns 93. No flags. Internal links: 3. External: 4.
  9. Publish gate → publish. JSON-RPC to WP. NitroPack purged. ClickUp task closed.
  10. T+7 tracker entry queued. Slack: "✅ Published: …".

Scenario B · GSC down (rare, but plan for it)

  1. Discover stage: GSC API returns 503. Adapter retries 3× with backoff. Still failing.
  2. Adapter falls back to Searchable's GSC-mirror data (degraded — less granular).
  3. Candidate pool drops from ~32 to ~22 (lose striking-distance precision).
  4. Pipeline continues normally with the smaller pool.
  5. Slack: "⚠️ GSC unavailable, used Searchable mirror — candidates: 22, published: 1".
  6. Next-day run automatically uses GSC again (no manual recovery).

Scenario C · Score <90, draft instead

  1. Discover → slate → brief succeeds.
  2. Draft writes, but focus keyword density lands at 0.8% (target 1–2.5%) AND only 1 external link present.
  3. Rank Math scorer returns 84. Two missing items.
  4. Publish gate → draft_queue. Saves as WP draft. ClickUp task → "Needs Review".
  5. Slack to Josh: "📝 Draft today (score 84/100): missing density + ext-link. Review at …".
  6. Next day's run not blocked — independent.
  7. Weekly Monday digest summarizes all draft-queue items for Josh's 5-min review.

Scenario D · Cannibalization conflict on every candidate (rare)

  1. Discover returns 32 candidates. Cannibalization layer drops 27 — recent runs have saturated the easy keywords.
  2. 5 candidates survive, but all are weak (priority <0.4 on the 0–1 scale).
  3. Orchestrator policy: if <5 strong candidates, skip the day rather than publish thin content.
  4. Slack to Josh: "🛑 No publish today — keyword pool depleted. Recommend refresh sprint instead of new posts." Includes the top 5 weak candidates for manual judgment.
  5. Discovery for tomorrow auto-broadens its window (60 days → 90 days impressions).

14. Risks + guardrails

Risk · auto-publish drift

A bug in scoring or brand-voice rubric could let bad content live before anyone notices. Guardrail: weekly Monday digest to Josh with all 7 posts of the prior week. First sub-3-star post triggers full pause until Josh reviews.

Risk · category saturation

If we publish 7×/week into only 4 categories, we'll saturate them in ~2 months. Guardrail: add the 3 secondary categories (AI in Business 13, AI Tools 14, AGI 10) at week 3 and after. Or switch to themed-day mode (Decision 11.3).

Risk · cannibalization with our own landing pages

"AI consulting for small business" overlaps the Tier-1 /ai-consulting-for-small-business/ page (priority page). Guardrail: inventory snapshot includes Tier-1 pages, not just blog posts. Cannibalization layer treats landing pages as "supreme" — a blog never competes with a landing page on the same focus keyword.

Risk · Anthropic API budget

30 days × Sonnet 4.6 ≈ $9–15/mo. Cheap. But Opus default would be $40–60/mo. Guardrail: hard monthly cap via tools/content/budget_tracker.py (already exists for v5). Stops publishing if cap exceeded; alerts Slack.

Risk · NitroPack stale cache

Even after publish, NitroPack can serve stale HTML for ~5 min if its purge fires too fast. Guardrail: the existing publish step already calls NitroPack purge twice with a 60s delay (proven pattern).

Risk · GSC rate limits

GSC API caps at ~1200 requests/min per project. Our usage: ~6 requests per discover run, daily. ~0.005% of the cap. Guardrail: none needed.
